# Web UI Action System v1

## 1. 목적

이 문서는 현재 웹 서비스 전반에 적용할 전역 UI 액션 시스템 v1 정의서다.
문제는 특정 카드의 버튼 모양이 아니라, 탐색 액션, 서비스 액션, 근거 액션, 정보 chip이 같은 무게로 섞여 보이는 데 있다.
후속 화면 구현자는 이 문서를 기준으로 어떤 행동을 가장 먼저 노출해야 하는지 판단한다.

## 2. 디자인 방향

- 도구형 70
- 에디토리얼형 30
- 핵심 목표는 “무엇이 클릭 가능한지”와 “다음에 무엇을 해야 하는지”를 빠르게 스캔하게 만드는 것이다.
- 장식성보다 위계, 밀도, 반복 가능성을 우선한다.

## 3. 액션 위계

1. 탐색 액션
2. 서비스 액션
3. 근거 액션
4. 상태/형식 정보

정의:
- 탐색 액션: 서비스 내부의 다음 화면이나 상세 맥락으로 이동
- 서비스 액션: Spotify, YouTube Music, YouTube MV 같은 외부 청취/시청 이동
- 근거 액션: 기사 원문, 공식 공지, 출처 확인
- 상태/형식 정보: 날짜 상태, 릴리즈 형식, confidence, classification tag

원칙:
- 한 카드에서 사용자가 가장 먼저 봐야 할 행동은 항상 탐색 액션이다.
- 서비스 액션은 중요하지만, 탐색 액션보다 먼저 시선을 빼앗으면 안 된다.
- 근거 액션은 신뢰 보강용이며, 핵심 CTA보다 항상 약해야 한다.
- 상태/형식 정보는 액션이 아니다.

## 4. 액션 타입

### 4.1 Primary

목적:
- 현재 카드나 섹션에서 가장 중요한 다음 행동

대표 예시:
- `팀 페이지`
- `앨범 상세`
- `릴리즈 상세`

규칙:
- 카드/섹션당 최대 1개
- 가장 강한 대비, 명확한 버튼 형태, 가장 높은 클릭 우선순위 사용
- 사용자가 얻는 결과를 바로 이해할 수 있는 라벨 사용
- 구현/기술 표현 대신 목적 중심 라벨 사용

금지:
- 같은 카드에 Primary 2개 이상 배치
- 정보 chip과 비슷한 형태로 만들기

### 4.2 Secondary

목적:
- Primary 다음 순위의 보조 탐색 행동

대표 예시:
- `앨범 상세`가 Primary인 카드의 `팀 페이지`
- `더보기`

규칙:
- 탐색 계열이지만 Primary보다 약해야 한다.
- 버튼이라는 사실은 유지하되, 시각적 비중은 한 단계 낮춘다.
- 없는 화면이 많아도 된다. Secondary가 억지로 필요하지는 않다.

### 4.3 Service

목적:
- 외부 음악/영상 서비스로 직접 이동

v1 대상:
- `Spotify`
- `YouTube Music`
- `YouTube MV`

규칙:
- 가능한 한 그룹으로 묶는다.
- 버튼 구조는 서비스마다 동일하고, 식별 요소는 아이콘 mark와 라벨에 한정한다.
- canonical 링크와 검색 fallback 차이는 버튼 표면보다 보조 정보, tooltip, 접근성 텍스트에서 처리한다.
- 서비스 액션은 탐색 액션보다 앞에 두지 않는다.

### 4.4 Meta

목적:
- 근거 확인, 원문 열기, 출처 검증

대표 예시:
- `기사 원문`
- `공식 공지`
- `출처 보기`

규칙:
- 가장 약한 시각적 비중 사용
- 텍스트 링크 또는 매우 가벼운 보조 버튼 수준 유지
- 서비스 액션보다 항상 뒤에 오고, 같은 줄에 놓여도 먼저 읽히지 않게 한다.

### 4.5 Informational Chip

목적:
- 상태, 형식, 분류를 요약하는 비액션 표시

대표 예시:
- `확정`
- `예정`
- `루머`
- `싱글`
- `EP`
- `타이틀`
- `confidence`

규칙:
- hover, outline, elevation, pointer affordance 때문에 버튼처럼 보이면 안 된다.
- 눌렀을 때 이동/열림이 없으면 버튼 모양을 쓰지 않는다.
- 정보군으로 묶되, CTA 영역과 섞지 않는다.

## 5. 전역 규칙

### 5.1 카드당 CTA 규칙

- 카드당 Primary 액션은 최대 1개
- Service 액션은 최대한 한 그룹으로 묶기
- Meta 액션은 별도 줄 또는 가장 아래 보조 영역에 배치
- informational chip은 CTA 줄에 섞지 않기

### 5.2 배치 순서

- 상단: identity + informational chip
- 본문: headline + 핵심 메타
- 하단 1순위: Primary 또는 Primary + Secondary
- 하단 2순위: Service group
- 하단 3순위: Meta action

### 5.3 라벨 규칙

- 구현 관점 표현 금지
- 사용자가 실제로 하게 되는 행동을 그대로 쓴다.
- `검색 열기`, `외부 링크`, `canonical`, `fallback` 같은 내부 표현은 버튼 라벨로 쓰지 않는다.
- 서비스 버튼은 `Spotify`, `YouTube Music`, `YouTube MV`처럼 브랜드/행동이 바로 읽히는 짧은 명사 라벨 사용
- 탐색 버튼은 `팀 페이지`, `앨범 상세`, `릴리즈 상세`처럼 도착 맥락이 명확해야 한다.

### 5.4 링크/버튼 선택 규칙

- 내부 화면 이동은 버튼형 control 우선
- 외부 서비스 이동은 service button system 사용
- 출처 확인은 text-link 또는 meta button 사용
- 동일 카드에서 text-link, button, chip이 같은 톤으로 보이면 실패로 본다.

## 6. 표면별 기준

### 6.1 캘린더 날짜 상세 카드

- Primary: `팀 페이지`
- Secondary: `릴리즈 상세` 또는 `앨범 상세`
- Service: `Spotify`, `YouTube Music`, `YouTube MV`
- Meta: `기사 원문`, `출처 보기`
- informational chip: 날짜 상태, 릴리즈 형식, confidence

설명:
- 사용자는 날짜를 눌렀을 때 가장 먼저 “이 팀을 더 볼지” 판단한다.
- 그래서 `팀 페이지`가 가장 강한 CTA여야 한다.

### 6.2 예정 스캔 카드

- Primary: `팀 페이지`
- Secondary: 기본적으로 없음
- Service: 기본 v1에서는 약하거나 생략 가능
- Meta: `기사 원문`, `공식 공지`, `출처 보기`
- informational chip: 상태, confidence, 분류 tag

설명:
- 예정 스캔은 탐색 보조 패널이므로, 근거 액션이 팀 탐색보다 강해지면 안 된다.

### 6.3 팀 페이지

- Primary: `최신 발매 상세`, `최근 앨범 상세`
- Secondary: 필요 시 `팀 페이지 내 다른 섹션 진입`
- Service: 최신 발매 기준 `Spotify`, `YouTube Music`, `YouTube MV`
- Meta: 공식 SNS, 아티스트 출처, 발매 출처
- informational chip: 트래킹 상태, 형식, 문맥 tag

설명:
- 팀 페이지 자체가 탐색 종착점 중 하나이므로, 여기서 가장 강한 다음 행동은 “특정 발매 맥락 진입”이어야 한다.

### 6.4 앨범/릴리즈 상세

- Primary: 없거나 매우 제한적
- Secondary: 팀 페이지 복귀 또는 인접 상세 이동
- Service: 앨범 레벨/트랙 레벨 `Spotify`, `YouTube Music`, `YouTube MV`
- Meta: 발매 출처, 아티스트 출처, 노트, 크레딧
- informational chip: 형식, 타이틀, special tag

설명:
- 상세 화면에서는 사용자의 핵심 목적이 감상 이동이므로 service group 비중이 커질 수 있다.
- 그래도 source link가 service보다 앞에 나오면 안 된다.

### 6.5 월간 대시보드

- Primary: `팀 페이지` 또는 `상세 보기`
- Secondary: 필요 시 보조 탐색
- Service: 압축된 service group
- Meta: 출처, 상태
- informational chip: 날짜 상태, 형식

설명:
- 리스트 밀도가 높은 화면이므로, CTA 위계가 더 엄격해야 한다.
- 서비스 버튼이 표 셀의 메인 텍스트처럼 보이면 실패다.

### 6.6 팀 디렉터리/검색 결과

- Primary: `팀 페이지`
- Secondary: 기본적으로 없음
- Service: 없음
- Meta: 필요 시 없음
- informational chip: 예정 신호 상태 정도만 허용

설명:
- 이 표면의 목적은 탐색 진입이다. CTA를 여러 개 두지 않는다.

## 7. 상태별 규칙

### 7.1 Empty

- 설명 문구 + 필요한 경우 Primary 1개만 노출
- empty 상태에서 Meta action은 기본적으로 숨긴다.

### 7.2 Partial Data

- 서비스 링크 일부만 있어도 service group 구조는 유지
- Meta 링크가 없으면 해당 항목만 숨긴다.
- 정보 누락 때문에 chip을 버튼처럼 대체하지 않는다.

### 7.3 External Fallback

- 서비스 검색 fallback은 버튼 라벨을 바꾸지 않는다.
- 차이는 접근성 문구, tooltip, 보조 설명, 내부 데이터 필드에서 처리한다.

### 7.4 Dense Layout

- 공간이 좁아져도 chip과 action을 한 줄에 뒤섞지 않는다.
- 줄 수가 부족하면 Meta부터 약화하거나 아래로 내린다.
- Primary를 희생해서 Service만 남기는 식의 축소는 금지한다.

## 8. 구현 판단 체크리스트

구현자는 카드/섹션마다 아래 질문을 순서대로 답해야 한다.

1. 사용자가 이 블록에서 가장 먼저 해야 할 행동은 무엇인가?
2. 그 행동이 탐색 액션인가, 서비스 액션인가, 근거 액션인가?
3. Primary를 하나만 두었는가?
4. Service 버튼이 하나의 그룹처럼 보이는가?
5. Meta 액션이 Service보다 약하게 보이는가?
6. chip이 버튼처럼 보이지 않는가?
7. 버튼 라벨이 내부 구현 언어가 아니라 사용자 행동 언어인가?

## 9. 후속 이슈 연결

- `#58`은 이 문서의 `Service` 정의를 구체적인 브랜드 버튼 시스템으로 시각화한다.
- `#59`는 이 문서의 표면별 기준을 실제 캘린더 상세, 예정 스캔, 팀 페이지, 앨범 상세, 월간 대시보드에 적용한다.

## 10. v1 수용 기준

- 후속 구현자가 각 화면에서 어떤 액션을 Primary로 둘지 즉시 판단할 수 있다.
- `Primary` / `Secondary` / `Service` / `Meta` / `Informational chip`의 경계가 문서만으로도 분명하다.
- informational chip이 버튼처럼 보이면 안 된다는 규칙이 명확히 적혀 있다.
- 서비스 액션이 탐색 액션보다 앞서지 않도록 배치 원칙이 명시돼 있다.
