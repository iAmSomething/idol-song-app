# Decision Log

## 1. 목적
이 문서는 모바일/제품 방향에서 이미 결정된 핵심 판단을 기록한다.
후속 구현 중 같은 논의를 반복하거나, 과거 결정을 뒤집는 실수를 줄이기 위함이다.

## 2. 결정 목록

### D-01 앱 내 직접 음원 재생 미지원
- 결정: v1 범위에서 앱 내 직접 오디오 재생은 하지 않는다.
- 이유: 외부 서비스 제약이 크고, 제품의 핵심은 발견/추적/이동이다.

### D-02 외부 서비스 handoff 우선
- 결정: Spotify / YouTube Music / YouTube MV로 이동하는 구조를 기본으로 한다.
- 이유: 구현 난이도와 정책 리스크 대비 사용자 가치가 높다.

### D-03 팀 페이지는 팬페이지가 아니라 실용 허브
- 결정: 팀 페이지 상단 순서는 `다음 컴백 -> 최신 발매 -> 최근 앨범`이다.
- 이유: 이 앱의 핵심 목적이 컴백 추적과 빠른 이동이기 때문이다.

### D-04 앨범 상세는 독립 가치가 큰 소비 화면
- 결정: 앨범/릴리즈 상세는 트랙 단위 handoff와 MV 확인을 제공한다.
- 이유: 팀 페이지만으로는 곡 단위 소비가 부족하다.

### D-05 모바일 날짜 drill-in은 bottom sheet
- 결정: 모바일에서 날짜 탭 결과는 bottom sheet로 연다.
- 이유: 캘린더 컨텍스트를 유지하면서 drill-in 하기 가장 적합하다.

### D-06 장르 판정 라벨 미노출
- 결정: K-pop / adjacent 같은 분류를 사용자에게 노출하지 않는다.
- 이유: 낙인처럼 보일 수 있고 제품 가치와 직접 관련이 없다.

### D-07 검색은 공식명 중심이 아니라 alias-aware
- 결정: 한글명, 약칭, 표기 변형을 search aliases로 지원한다.
- 이유: 실제 사용자 검색 행태를 반영해야 하기 때문이다.

### D-08 태그는 액션이 아님
- 결정: status/format/title chip은 절대 버튼처럼 보이면 안 된다.
- 이유: 현재 UI 혼동의 주요 원인이 태그와 액션의 혼재다.

### D-09 레이더의 주 CTA는 팀 페이지
- 결정: Radar에서는 음악 서비스 액션보다 Team Detail 진입이 우선이다.
- 이유: Radar는 탐지/판단 허브이지 소비 화면이 아니다.

### D-10 최신곡과 최신앨범은 분리 유지
- 결정: `latest_song`, `latest_album`을 별개로 관리한다.
- 이유: 선공개곡/타이틀/앨범 발매 간 시차를 정확히 반영해야 한다.

## 3. 사용 규칙
- 새 큰 결정이 나오면 이 문서에 추가한다.
- 구현이 이 결정을 뒤집을 경우, 먼저 문서를 업데이트하고 이유를 남긴다.
