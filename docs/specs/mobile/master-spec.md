# Mobile Master Spec

## 1. 문서 목적

이 문서는 `idol-song-app`의 차후 모바일 앱 구현을 위한 상위 제품 명세서다.
목표는 Android/iOS 공용 앱을 `Expo + React Native` 기준으로 설계하되, 현재 웹에서 검증 중인 핵심 가치인 `발견`, `추적`, `이동`을 모바일 UX에 맞게 재구성하는 것이다.

이 문서는 다음을 결정한다.
- 모바일 정보 구조(IA)
- 전역 내비게이션 방식
- 화면 간 이동 규칙
- 공통 액션 위계
- 데이터 재사용 전략
- 구현 단계와 선행 조건

## 2. 제품 목표

### 2.1 핵심 목표
- 사용자는 월간 발매/컴백 일정을 빠르게 훑을 수 있어야 한다.
- 사용자는 특정 날짜의 발매/예정 정보를 한 번의 탭으로 확인할 수 있어야 한다.
- 사용자는 팀 페이지와 릴리즈 상세로 빠르게 이동할 수 있어야 한다.
- 사용자는 Spotify, YouTube Music, YouTube MV로 즉시 이동할 수 있어야 한다.
- 사용자는 팀/발매/예정을 별칭과 한글명으로 검색할 수 있어야 한다.

### 2.2 비핵심 목표
- 앱 내 직접 오디오 재생
- 플레이리스트 생성
- 팬 커뮤니티 기능
- 완전한 SNS 소비 앱화

## 3. 사용자 유형

### 3.1 파워 청취자
- 많은 팀의 발매를 놓치지 않으려 한다.
- 월간 캘린더와 필터를 자주 쓴다.
- 팀 페이지보다 `날짜 -> 상세 -> 외부 서비스 이동` 흐름을 자주 쓴다.

### 3.2 팀 중심 탐색자
- 특정 팀의 다음 컴백과 최신 발매를 보고 싶어 한다.
- 팀 상세와 앨범 상세를 자주 쓴다.

### 3.3 레이더 사용자
- 가장 가까운 컴백, 일정 변경, 장기 공백 팀을 빠르게 보고 싶어 한다.
- 레이더 탭을 요약 대시보드처럼 사용한다.

## 4. 플랫폼 방향

### 4.1 기술 가정
- 프레임워크: Expo + React Native
- 내비게이션: 탭 + 스택 + 시트 조합
- 데이터 소스: 현재 웹의 정적 JSON 산출물 재사용 우선

### 4.2 공유 대상
- 타입 / 데이터 계약
- alias 검색 정규화
- source / confidence 규칙
- release kind 분류
- handoff 링크 생성기
- 날짜 정렬 및 grouping 규칙

### 4.3 재설계 대상
- 레이아웃
- 컴포넌트 구조
- 시트 인터랙션
- 탭 구조
- 화면 간 전환 UX

## 5. 정보 구조 (IA)

### 5.1 하단 탭
1. `캘린더`
2. `레이더`
3. `검색`

### 5.2 스택 화면
1. `팀 상세`
2. `앨범/릴리즈 상세`

### 5.3 전역 시트
1. `날짜 상세 bottom sheet`
2. `필터 시트`
3. `언어/설정 시트`

## 6. 전역 내비게이션 규칙

### 6.1 기본 진입
- 앱 첫 진입은 `캘린더 탭`으로 고정한다.
- 현재 월 기준 캘린더가 보인다.

### 6.2 탭 전환
- 하단 탭은 상태를 보존한다.
- 같은 탭 재탭 시 상단으로 스크롤하거나 기본 상태로 복귀하는 동작은 v2 고려 사항으로 둔다.

### 6.3 상세 이동
- `팀 상세`, `앨범 상세`는 push 전환으로 연다.
- 뒤로 가기는 네이티브 back 규칙을 따른다.

### 6.4 날짜 상세
- 날짜 탭 시 별도 페이지 이동 없이 bottom sheet를 연다.
- sheet를 닫으면 캘린더 컨텍스트로 복귀한다.

## 7. 전역 UX 원칙

### 7.1 레이아웃
- 큰 히어로 섹션은 사용하지 않는다.
- 상단은 월/탭/검색/필터 중심의 도구형 헤더를 사용한다.
- 화면당 핵심 목적을 하나만 강조한다.

### 7.2 액션 위계
1. 탐색 액션 (`팀 페이지`, `상세 보기`)
2. 서비스 액션 (`Spotify`, `YouTube Music`, `MV`)
3. 출처 액션 (`기사`, `공식 공지`, `소스 보기`)
4. 상태 태그는 액션이 아니다.

### 7.3 스캔성
- 카드 하나에 행동 버튼이 과도하게 많아지지 않게 한다.
- 태그는 요약용, 버튼은 이동용으로 명확히 구분한다.
- 서비스 버튼은 브랜드 식별이 가능해야 한다.

## 8. 데이터 계약

모바일은 아래 산출물을 읽는 구조를 우선 검토한다.
- `web/src/data/releases.json`
- `web/src/data/watchlist.json`
- `web/src/data/upcomingCandidates.json`
- `web/src/data/artistProfiles.json`
- `web/src/data/releaseArtwork.json`
- `web/src/data/releaseDetails.json`

각 화면은 실시간 API 호출보다 위 정적 데이터 계약을 우선 소비한다.

## 9. 화면 트리

### 9.1 캘린더 탭
- 월간 캘린더
- 월간 요약
- 리스트 모드 세그먼트
- 날짜 상세 sheet

### 9.2 레이더 탭
- 가장 가까운 컴백
- 이번 주 예정
- 일정 변경
- 장기 공백 레이더
- 루키 레이더

### 9.3 검색 탭
- 통합 검색창
- 결과 세그먼트 (`팀`, `발매`, `예정`)
- 최근 검색

### 9.4 팀 상세
- 헤더
- 다음 컴백
- 최신 발매
- 최근 앨범

### 9.5 앨범/릴리즈 상세
- 커버/메타
- 앨범 레벨 서비스 액션
- 트랙 리스트
- MV 영역

## 10. 주요 유스케이스

### UC-01 이번 달 일정 확인
- 사용자는 캘린더 탭에 진입한다.
- 현재 월의 발매/예정 현황을 확인한다.
- 특정 날짜를 탭한다.
- 상세 sheet에서 팀 페이지 또는 서비스 링크로 이동한다.

### UC-02 이번 주 컴백 확인
- 사용자는 레이더 탭에서 `이번 주 예정` 섹션을 본다.
- 관심 팀을 탭한다.
- 팀 상세에서 최신 발매와 다음 컴백을 확인한다.

### UC-03 별칭 검색
- 사용자는 검색 탭에서 `투바투`, `트와이스`, `블핑` 같은 별칭을 입력한다.
- 결과 세그먼트를 전환하며 팀, 발매, 예정 결과를 확인한다.
- 원하는 결과로 이동한다.

### UC-04 앨범 수록곡 이동
- 사용자는 팀 상세에서 최근 앨범 카드로 들어간다.
- 앨범 상세에서 트랙 리스트를 본다.
- 특정 트랙의 Spotify 또는 YouTube Music 버튼을 탭한다.

## 11. 애니메이션 원칙

### 11.1 허용 애니메이션
- 탭 전환 시 기본 네이티브 전환
- push/pop 시 기본 네이티브 전환
- bottom sheet open/close
- 카드/필터 적용 시 짧은 fade/translate

### 11.2 피해야 할 것
- 과한 패럴랙스
- 대형 히어로 모션
- 서비스 버튼 개별 과장 애니메이션

### 11.3 강도
- 짧고 명확한 피드백 중심
- 정보 탐색을 방해하지 않음

## 12. 구현 단계

### Phase 0
- 웹 계속 검증
- 모바일은 문서와 데이터 계약 정리

### Phase 1
- 공유 가능한 도메인/유틸 로직 분리
- 모바일 데이터 계약 정의

### Phase 2
- Expo 앱 초기화
- 탭/스택/시트 기본 구조 구현

### Phase 3
- 캘린더 탭
- 팀 상세
- 앨범 상세

### Phase 4
- 검색 탭
- 레이더 탭

### Phase 5
- 모바일 전용 가치 확장
  - 푸시 알림
  - 공유
  - 딥링크 강화

## 13. 선행 조건
- 웹 팀 페이지 구조 확정
- 웹 앨범 상세 구조 확정
- alias 검색 전략 확정
- `releaseDetails` / `releaseArtwork` / `artistProfiles` 데이터셋 안정화
- upcoming / hydration 운영 전략 확정

## 14. 비범위
- 지금 당장 모바일 앱 생성
- iOS/Android 배포 파이프라인 구축
- 앱 내 스트리밍 재생
- 웹 UI를 RN으로 그대로 치환

## 15. 비기능 요구사항
- 화면 첫 진입 후 핵심 정보는 긴 소개문 없이 바로 보여야 한다.
- 서비스 버튼과 출처 링크는 시각적으로 명확히 구분되어야 한다.
- 탭/시트/상세 전환은 상태를 보존해야 한다.
- 긴 텍스트, 누락 이미지, 누락 링크, 모호한 날짜를 정상 상태로 처리해야 한다.
- 접근성, localization, visual token 규칙은 각 개별 화면보다 상위 계약으로 취급한다.
- 데이터 신선도, 외부 링크 보안, 분석 이벤트 규칙은 별도 하위 문서를 따른다.

## 16. 수용 기준
- 이 문서만 읽고 모바일 앱의 최상위 구조를 이해할 수 있어야 한다.
- 하단 탭, 상세 흐름, 데이터 재사용 전략, 구현 단계가 명확해야 한다.
- 후속 화면 문서를 작성하거나 개발 이슈로 분해할 수 있어야 한다.
